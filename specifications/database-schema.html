Entity overview

  • admin — admin profile attached 1:1 to a user
  • earned_experience — an immutable XP grant (workout completion or claimed quest), locked by the nightly cron
  • equipment — admin-managed reference (barbell, dumbbell, …)
  • exercise — one movement inside a workout, ordered by position
  • exercise_set — one set inside an exercise, with planned/achieved metrics
  • hydration_daily_summary — one row per player per day; snapshotted target + summed intake
  • hydration_entry — a single drink logged against a day's hydration summary
  • level_bracket — one contiguous slice of the leveling curve (marginal XP cost a·n^k + b)
  • leveling_config — singleton of admin-editable leveling parameters (xp_per_workout_minute)
  • movement — admin-managed exercise definition (slug, label, main muscle, tracking flags)
  • movement_equipment — M:N join, equipment options per movement
  • movement_secondary_muscle — M:N join, secondary muscles per movement
  • muscle — admin-managed reference (biceps, triceps, …)
  • personal_best — current best for a (player, movement, type) tuple
  • player — player profile attached 1:1 to a user
  • quest — admin-defined challenge (automatic metric-tracked or manual), with a reward and an active window
  • quest_progression — one player's progress on a quest for a given period (find-or-created lazily)
  • sleep_daily_entry — one sleep record per player per night (wake-up date)
  • steps_daily_entry — one step count + goal per player per day
  • user — accounts (auth credentials, roles)
  • weight_entry — one body-weight measurement per player per day
  • workout — a planned, in-progress, completed or canceled training session

Relationships at a glance

required FK (one-to-many / one-to-one) optional / nullable FK many-to-many (via join table)

The movement → personal_best link (one PB per movement, per player, per type) is shown by the FK list inside the personal_best box rather than a separate arrow, to keep the diagram readable. Likewise, every Tracking entity is N:1 to player (one row per player per day, except hydration_entry which is N:1 to its hydration_daily_summary); these player FKs are noted inside each box instead of drawn as crossing arrows. In the Leveling lane, earned_experience is N:1 to player (FK noted in-box); level_bracket (the XP curve) and the leveling_config singleton carry no foreign keys. In the Questing lane, quest is admin-defined reference data (no FK), and quest_progression is N:1 to both quest and player (the player FK is noted in-box; the quest link is drawn).

admin

An admin profile, attached 1:1 to exactly one user account (with ROLE_ADMIN on the user). Carries the HR fields needed to manage admin staff: first/last name, job title, hire date, and an active/inactive status (so that the account of an admin who leaves the company can be deactivated without being deleted). Created via App\Infrastructure\Persister\User\AdminPersister::create(RegisterAdminDataInput), which seeds the user with the admin role and the admin profile in one call. status defaults to ACTIVE on creation — see AdminStatusRegistry.

level_bracket

One contiguous slice of the leveling curve: for any level n in [from_level, to_level] (to_level = NULL means open-ended, last bracket), the marginal XP cost of the next level is coefficient_a · n^exponent_k + offset_b. The admin CRUD validators enforce contiguity, a single open-ended bracket, and from_level = 1 on the first row. Seeded by Version20260613120000 (3 baseline rows).

quest

An admin-defined challenge. kind = AUTOMATIC quests track a metric toward a target_value (both non-null — see QuestMetricRegistry); kind = MANUAL quests carry no metric/target and are claimable directly. periodicity (UNIQUE / DAILY / WEEKLY / MONTHLY) decides how often a fresh quest_progression is materialised per player. The quest is active for any instant in [date_start, date_end] (date_end = NULL = open-ended). The admin CRUD validators enforce the kind ⇔ metric ⇔ target_value coupling, rewarded_xp > 0, and date_end > date_start.
