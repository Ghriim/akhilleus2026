Entity overview

  • admin — admin profile attached 1:1 to a user
  • equipment — admin-managed reference (barbell, dumbbell, …)
  • exercise — one movement inside a workout, ordered by position
  • exercise_set — one set inside an exercise, with planned/achieved metrics
  • hydration_daily_summary — one row per player per day; snapshotted target + summed intake
  • hydration_entry — a single drink logged against a day's hydration summary
  • movement — admin-managed exercise definition (slug, label, main muscle, tracking flags)
  • movement_equipment — M:N join, equipment options per movement
  • movement_secondary_muscle — M:N join, secondary muscles per movement
  • muscle — admin-managed reference (biceps, triceps, …)
  • personal_best — current best for a (player, movement, type) tuple
  • player — player profile attached 1:1 to a user
  • sleep_daily_entry — one sleep record per player per night (wake-up date)
  • steps_daily_entry — one step count + goal per player per day
  • user — accounts (auth credentials, roles)
  • weight_entry — one body-weight measurement per player per day
  • workout — a planned, in-progress, completed or canceled training session

Relationships at a glance

required FK (one-to-many / one-to-one) optional / nullable FK many-to-many (via join table)

The movement → personal_best link (one PB per movement, per player, per type) is shown by the FK list inside the personal_best box rather than a separate arrow, to keep the diagram readable. Likewise, every Tracking entity is N:1 to player (one row per player per day, except hydration_entry which is N:1 to its hydration_daily_summary); these player FKs are noted inside each box instead of drawn as crossing arrows.

Tables

admin

An admin profile, attached 1:1 to exactly one user account (with ROLE_ADMIN on the user). Carries the HR fields needed to manage admin staff: first/last name, job title, hire date, and an active/inactive status (so that the account of an admin who leaves the company can be deactivated without being deleted). Created via App\Infrastructure\Persister\User\AdminPersister::create(RegisterAdminDataInput), which seeds the user with the admin role and the admin profile in one call. status defaults to ACTIVE on creation — see AdminStatusRegistry.
